前车之覆，后车之鉴。

​ ——荷兰谚语

如果我们想解决问题，就必须试图先去了解问题。

编程系统产品

程序

程序：程序本身是完整的，可以由作者在所开发的系统上运行。它通常是车库中产出的产品，以及作为单个程序员生产率的评估标准。

有两种途径可以使程序转变成更有用但是成本更高的产物：

编程产品

编程产品：编程产品可以被任何人运行、测试、修复和扩展的程序。它可以在多种操作系统平台上运行，供多套数据使用。

程序成为编程产品的条件：
  • 程序必须按照普遍认可的风格来编写，特别是输入的范围和形式必须广泛地适用于所有可以合理使用的基本算法。
  • 对程序进行彻底测试，确保它的稳定性和可靠性，使其值得信赖。这就意味着必须准备、运行和记录详尽的测试用例库，用来检查输入的边界和范围。
  • 要有完备的文档。每个人都可以对其加以使用、修复和扩展。

编程系统构件

编程系统构件：编程系统构件是在功能上能相互协作、具有规范的格式、可以进行交互的程序集合，并可以用来组装和搭建整个系统。

程序成为编程系统构件的条件：

  • 程序必须按照一定的要求编制，使输入和输出在语法和语义上与精确定义的接口一致。

  • 同时程序还要符合预先定义的资源限制——内存空间、输入输出设备、运行时间。

  • 程序必须同其他系统构件一起以任何能够想象到的组合进行测试。

    ​ 由于测试用例会追随着组合组合不断增加，所以测试的范围必须广泛。因为一些意象不到的交互会产生许多不易察觉的bug，测试工作会非常耗时，因此相同功能的编程系统构件的成本至少是独立程序的3倍。如果系统有大量的组成单元，成本还会更高。

编程系统产品

与以上的所有简单的程序都不同的是，它的成本高达9倍。然而，只有它才是真正有用的产品，是大多数系统开发的目标。

编程的乐趣

  • 编程的快乐是一种创建事物的纯粹快乐。
  • 编程的快乐来自于开发对他人有用的东西。
  • 编程的快乐来自于整个过程体现出一股强大的魅力——将相互独立的零部件组装在一起，看到它们以精妙的方式运行着，并收到了预期的效果。
  • 编程的快乐是持续学习的快乐，它来自于这项工作的非重复性。
  • 编程的快乐还来自于易于驾驭的介质上工作。
  • 编程的快乐在于它不仅满足了我们内心深处进行创造的渴望，而且还唤醒了每个人内心的情感。

职业的苦恼

  • 首先，苦恼来自追求完美。学习编程最困难的部分是将做事的方式向追求完美的方向调整。

  • 苦恼来自由他人来设定目标、供给资源和提供信息。

    ​ 对于系统编程人员而言，对其他人的依赖是一件非常痛苦的事情。他依靠其他人的程序，而这些程序往往设计得并不合理、实现拙劣、发布不完整（没用源代码或测试用例）或者文档记录得很糟糕。所以，系统编程人员不得不花费时间去研究和修改，而它们在理想情况下本应该是可靠的、完整的。

  • 概念性设计是有趣的，但寻找琐碎的bug却是一项重复性的活动。

    另外，人们发现调试和查错往往是线性收敛的，或者更糟糕的是具有二次方的复杂度。

  • 当投入了大量辛苦的劳动，产品在即将完成或者终于完成的时候，却已显得陈旧过时。

第一章读后感

程序和能够使用的产品之间确实还有很大的距离，当一个程序被设计出来使我们往往只考虑了理想的输入，拥有我们想要的一切运行环境，而我们的程序真正被别人所使用的时候，这一切都不一定拥有，使用者可能有着稀奇古怪的输入，而且使用者大概率并没有我们的编程环境，而且对于在不同计算机平台上能否成功运行也是一个需要考虑的问题，所以在程序完成后，测试、打包项目如同编写程序一样重要，因为没有后续的部分，你编写出来的程序只是一堆只能有你一个人使用的代码。还有说明文档也是必不可少的环节，之前如何写 readme 一直是我的一大困惑，现在虽然还不能写的很好，但是已近能够基本对程序进行说明了。

对于编程的乐趣和苦恼部分我觉得作者总结的很正确，的确如此编程是一件痛并快乐着的事情，一个bug找半天也找不出来的时候确实让人头疼，但是程序终于如期运行的时候那种愉悦感也是别的事情无法替代的。
